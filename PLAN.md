# In-app Comment Tool 仕様書

## 1. 概要

dev 環境上で動作する Vue アプリに対し、社内の非エンジニア（PM・デザイナー・QA など）が **コンポーネントやページに直接コメントを残し、チームで共有・議論する** ための軽量ツール。Figma のコメント機能のページ版に相当する体験を目指す。

コメントは「ローカルで AI に指示を出すときの一次情報」として使うことを想定しており、本ツール自体は AI に接続しない。

## 2. ゴール / ノンゴール

**ゴール**

- ページ上の任意の位置にピンを打ち、コメントとリプライを残せる
- コメントは社内ユーザー間で共有・リアルタイムに近い形で更新される
- 非エンジニアがブラウザ拡張なしで使える（DevTools を開かせない）
- シンプルな構成で完結し、社内インフラ承認を必要としない

**ノンゴール**

- AI 連携（コメント生成・要約等）
- 通知（Slack / メール）
- 本番環境への展開
- 権限管理（社内＝全員編集可で十分）
- ピクセル単位の正確なスクリーンショット添付

## 3. アクセス前提

- dev 環境は Cloudflare Tunnel 配下にあり、ingress は社内に限定済み
- すべての利用者は Clerk によりログイン済み
- 上記により「外部からアクセスされる前提」を排除でき、認可は Clerk JWT による本人確認のみで十分

## 4. アーキテクチャ

```
Vue app (dev env / CF Tunnel配下)
  └─ Vite plugin が条件付きでオーバーレイを注入
       └─ ピン操作・コメント UI を担当
            ↓ fetch + Clerk JWT
       Cloudflare Worker (個人アカウント)
            ↓ JWKS で JWT 検証 → 認可
            ↓ SQL 実行
       Cloudflare D1
```

**判断のポイント**

| 論点               | 採用                                                   | 不採用                     | 理由                                                      |
| ------------------ | ------------------------------------------------------ | -------------------------- | --------------------------------------------------------- |
| 表示方法           | ページ内オーバーレイ                                   | Vue Devtools Plugin API    | 非エンジニアに拡張インストール／DevTools 起動を要求しない |
| データ層           | Cloudflare D1 + Worker                                 | Supabase / 社内 GCP        | 個人アカウントで完結・GCP 承認不要                        |
| 認証               | Clerk JWT を Worker で検証                             | Worker 独自セッション      | 既存 Clerk ログインを再利用                               |
| Realtime           | ポーリング                                             | WebSocket / Durable Object | 遅延数秒で十分。後から差し替え可能                        |
| コンポーネント特定 | `vite-plugin-vue-inspector` の `data-v-inspector` 属性 | ランタイム探索のみ         | ファイルパス＋行番号が確定的に取れる                      |

## 5. 機能要件

### 5.1 ピン作成

- キーバインドで UI オーバーレイを ON にする。オーバーレイ表示中は以下が常に有効:
  - マウス hover 中、対象コンポーネントを視覚的にハイライト（参考: Vue Devtools の Inspector、Stagewise）
    - コンポーネント矩形にアウトライン + 薄いオーバーレイ
    - 近傍にラベル：コンポーネント名・ファイルパス・行番号
    - ラベル内に **GitHub の該当行へのリンク**を併設（ピンを打つ前から参照可能）
  - クリックでピン配置（コメント入力 UI を表示）。同一ページに複数ピンを置ける
  - 既存ピンの表示
- ネスト時はデフォルトで最内側コンポーネント。修飾キー（例: `Alt`）で親に遡れる
- **修飾キー**:
  - `Shift+click`: Finder 風の複数コンポーネント追加選択。コメント入力中にさらに紐づくコンポーネントを足せる（プライマリは最初に click したもので不動）
  - `Cmd+click`（macOS）/ `Ctrl+click`（Win/Linux）: 該当コンポーネントの GitHub 行を新規タブで開く
  - `Alt`: 親コンポーネントへ遡る
- **ページ全体宛てコメント**：オーバーレイ内の専用ボタン（スレッド一覧パネル等の UI 上）から開く。コンポーネント情報は持たず route + URL のみで紐付く

### 5.2 コメント / リプライ

- 1 ピン＝1 スレッド。スレッドに複数コメントがぶら下がる構造。1 スレッドにはプライマリコンポーネント 1 個に加え、`shift+click` で追加した「紐づくコンポーネント」（context references）を任意個持てる
- リプライは時系列で表示
- スレッドの resolve / unresolve トグル
- 自分のコメントは**編集・削除どちらも可**（他人のものは不可）
- 削除は soft delete。本文は「[削除されたコメント]」プレースホルダに置換し、リプライ文脈を保つ

### 5.3 表示

- UI オーバーレイが ON のとき、以下が表示される:
  - 既存ピン
  - 現在ページ上のスレッド一覧パネル（オーバーレイ内の UI で開閉）
  - コンポーネントごとの未読/件数バッジ
- resolved スレッドはデフォルト非表示。**一覧パネル内の UI トグル**で表示でき、履歴として常に参照可能
- 各スレッドおよび一覧パネルに **GitHub の該当行へのリンク**（スレッド保存時の commit hash 基準）を表示

### 5.4 ユーザー識別

- Clerk JWT の `sub` を user ID、カスタムクレームの `name` を表示名として使用
- `name` 取得失敗時のみフォールバックで入力プロンプト

### 5.5 コンテキスト保持

- 各スレッドに `route` パターン・実 URL・viewport サイズ・**ビルド時の commit hash**・**dirty フラグ**を保存
- commit hash は Vite ビルド時に `git rev-parse HEAD` を読み取り `import.meta.env` 経由で注入。スレッド作成時にスナップショットとして固定する
- AI への貼り付け用途のため、十分な周辺情報を持たせる
- **データは無期限保持**。アーカイブ・自動削除は行わない

## 6. データモデル

`threads`（ピン1個＝1レコード）と `comments`（リプライ含むコメント本体）の 2 テーブル構成。詳細スキーマは実装側で確定するが、以下を必ず含めること：

- **threads**: route, url, component_path **(nullable)**, component_line **(nullable)**, **commit_hash**, **dirty_build (bool)**, **x_ratio, y_ratio**, viewport_w/h, **additional_components (JSON array `[{path, line, index}, ...]`、nullable)**, status, created_by(Clerk sub), created_by_name
- **comments**: thread_id, body, created_by, created_by_name, **updated_at**, **deleted_at (soft delete)**

component_path / component_line が NULL のスレッドは「ページ全体宛て」として扱う。一覧パネルでも別グループ表示する。

comments は **soft delete** を採用：`deleted_at` を立て、本文は表示用プレースホルダ（例: "[削除されたコメント]"）に置換。リプライツリーの文脈が壊れないようにする。**ただしスレッド先頭コメント（最初の1件）が削除された場合はスレッドごと削除**し、リプライも合わせて消える（`ON DELETE CASCADE`）。

座標は **コンポーネント矩形に対する比率（0.0〜1.0）** で持つ。ページ絶対座標やピクセル値で持つと解像度・コンテンツ変動で破綻する。

## 7. 注意すべき設計トピック

### 7.1 コンポーネント識別の安定性

ファイルパス＋行番号で識別するため、**リファクタで対象が消失したスレッドは「stale」状態として右側パネルに浮かせる**フォールバックが必須。サイレントに消すと知見が失われる。

### 7.2 同一コンポーネントの複数インスタンス

ループや一覧表示で同じコンポーネントが複数回描画されるケース。`data-v-inspector` の行番号だけでは区別できない。**DOM 上の出現順インデックス**を併用して識別すること。

### 7.3 動的ルートと複数 dev URL

- `/users/123` と `/users/456` は同一 route（`/users/:id`）として扱う。`route` パターンと実 `url` の両方を保存し、デフォルトはパターン単位、URL 完全一致のフィルタも可能にしておく
- **PR preview など複数の dev URL がある場合も、route が同じなら横断的にコメントを共有する**。これにより「main の dev でついたコメントが preview 環境でも見える」体験になる
- 副作用：あるブランチでだけ存在するルートに対するコメントは、別ブランチで stale 扱いになる。これは 7.1 のフォールバックと同一の仕組みで吸収する

### 7.4 viewport 差異

PC とモバイルでレイアウトが大きく変わる UI ではピン位置が無意味になる。`viewport_w` を記録し、**現在 viewport と大きく異なるピンは半透明表示＋警告**するなどの逃げ道を用意。

### 7.5 認可は Worker コードで担保

D1 には Supabase のような RLS が存在しない。**全エンドポイントで JWT 検証を漏らさないこと**。ミドルウェアで一括適用し、認証チェック忘れによる露出を構造的に防ぐ。

### 7.6 CORS

Worker は dev 環境とは別ドメインに置かれるため、**origin の許可リストを Worker 側で明示**する必要がある。Clerk Dashboard 側の Allowed Origins にも dev ドメインを追加。

### 7.7 データ分離

社内本番プロダクトのデータベースとは完全に別の D1 プロジェクトに置く。

### 7.8 Commit hash の扱い

- ビルド時に Vite plugin で `git rev-parse HEAD` を環境変数として注入し、スレッド作成時にスナップショット保存。**後でコードが進んでもピン時点の正しい行**にリンクできる
- 未コミット変更がある dev ビルドでは hash が実コードと乖離する。`git status --porcelain` で dirty 判定し、**dirty フラグもスレッドに保存**。UI に「local changes」バッジを出して誤読を防ぐ運用とする（dirty build 自体は禁止しない）
- GitHub URL の組み立ては `https://github.com/<org>/<repo>/blob/<hash>/<component_path>#L<line>` の固定パターン。リポジトリ情報は env で持つ

### 7.9 オーバーレイ表示中の実装上の注意

- ハイライト用オーバーレイは `pointer-events: none` でビジュアル専用に。クリックは `document` レベルで capture phase に取る
- iframe・Shadow DOM 内コンポーネントは hover 検出のスコープ外として明示
- z-index：アプリ内モーダルより前面に出る必要があるが、自前モーダル（コメント入力 UI）はさらに上のレイヤに分ける
- hover 追従は `mousemove` を rAF スロットルしないと過剰再描画になる
- `data-v-inspector` 属性は本番ビルドでは付かない。**本ツールが有効な環境では vite-plugin-vue-inspector を必ず併用**することを前提とする
- **非 Vue 要素を hover した場合は最近接の Vue 親コンポーネントへフォールバック**する。`el.__vueParentComponent` を遡って最初に見つかった Vue コンポーネントを対象とする。これにより「テキスト直接」「素の div」へも実用的に pin できる

## 8. キーバインド

- **オーバーレイのトグル**: 単一の専用キーバインドのみ用意する（コンフリクトしにくく覚えやすいキーを選定）
- **オーバーレイ表示中のクリック修飾**:
  - 修飾なし: ピンを置きコメント入力 UI を開く
  - `Shift+click`: 紐づくコンポーネントを追加（複数選択）
  - `Cmd/Ctrl+click`: GitHub の該当行を新規タブで開く
  - `Alt+click/hover`: 親コンポーネントへ遡る
- 一覧パネルの開閉、resolved スレッドの表示/非表示、ページ全体宛てコメントの追加など、その他すべての操作はオーバーレイ内の UI で行う
